## 1. מטרה

סקיל Claude Code שמוביל בעל עסק **לא-טכני** דובר עברית מרעיון ועד אפליקציה עסקית חיה באוויר (CRM, ניהול לקוחות, דשבורד מדדים) — בלי שנתקע אף פעם, בלי שנחשוף נתונים של העסק שלו, ועם תוצאה שנראית מקצועית.

הסקיל ישותף עם תלמידי הסדנאות של טל בשור.

**עקרון הבסיס (נשמר מהמקור):** "אני עושה, אתה מחליט" — Claude מבצע הכול; המשתמש מחליט מה האפליקציה עושה ואיך היא נראית.

**סטאק (נשמר מהמקור):** React (Vite) + Supabase + Tailwind CSS v4 + Netlify. הכול בעברית.

## 2. זהות הסקיל

- **שם:** `business-app-builder` — "בונה אפליקציות עסקיות"
- **מבנה:** קובץ `SKILL.md` יחיד. בלי תיקיות עזר, בלי סקריפטים, בלי תלות בסקילים אחרים. יעד גודל: עד ~450 שורות, כתיבה הדוקה.
- **טריגרים:** כל הטריגרים של המקור ("תבנה לי אפליקציה", "בנה לי CRM", "אני רוצה דשבורד"...) + חדשים: "תמשיך לבנות את האפליקציה", "איפה עצרנו", "האפליקציה לא עובדת", "תוסיף פיצ'ר לאפליקציה".
- **התקנה:** מחליף את `fullstack-app` — המקור יוסר מ-`~/.claude/skills/` כדי למנוע התנגשות טריגרים.
- **קרדיט:** שורה בתחתית הקובץ: `Based on fullstack-app by Asher-pro (MIT). Redesigned and extended by Tal Bsore.`
- **שפת השיטה:** שאלות האפיון מנוסחות במונחים של טל — סיסטם, צוואר בקבוק, מדדים — בלי תלות בסקילים אחרים של הסדנה.

## 3. חוקי ברזל (בלוק MUST/NEVER בראש הסקיל)

1. **לעולם לא מדלגים על שער.** בין כל שני שלבים יש בדיקה אוטומטית; לא ממשיכים עד שהיא עוברת.
2. **לעולם לא בונים כפול.** אם קיים קובץ מצב — ממשיכים ממנו. אף פעם לא scaffold מחדש על פרויקט קיים.
3. **אבטחה לא נפתחת כדי "לתקן" תקלה.** בעיית הרשאות מאבחנים — לא פותרים בפתיחת ההגנות (בהשראת "Rationalizations to Reject" של Trail of Bits).
4. **עצירות חובה (נשמר מהמקור):** יצירת חשבונות, הקלדת סיסמאות, פרטי תשלום — רק המשתמש.
5. **מפתחות לעולם לא מודפסים** — לא לצ'אט, לא לקובץ מצב, רק ל-`.env`.
6. **"באוויר" רק עם הוכחה** — אין הכרזת הצלחה בלי בדיקה אמיתית על האתר החי.
7. **JavaScript פשוט בלבד** — בלי TypeScript, בלי ספריות state, קוד שמפתח עתידי יתחזק (נשמר מהמקור).
8. שגיאה שנכשלה פעמיים → **מעבר לדרך חלופית**, לא דיבוג מול המשתמש.

## 4. שלד: חמישה שלבים + שערים

השלד של המקור נשמר. בין השלבים נוספים שערים אוטומטיים (Claude מריץ לבד, המשתמש רואה ✔ בעברית):

| מעבר               | השער                                                              |
| ------------------ | ----------------------------------------------------------------- |
| אפיון → Supabase   | המפרט אושר ונכתב לקובץ המצב                                       |
| Supabase → קוד     | בדיקת round-trip מול כל טבלה עם מפתח anon (תקינות חיבור + הרשאות) |
| קוד → בדיקה מקומית | `npm run build` נקי + וידוא ש-Tailwind באמת מוחל על הפלט          |
| בדיקה → העלאה      | שרת פיתוח מחזיר 200 על דף הבית ועל עמוד עמוק, בלי שגיאות בלוג     |
| העלאה → סיום       | שער אבטחה E0 + smoke test מלא על ה-URL החי                        |

### קובץ מצב והמשכיות

- אחרי אישור המפרט נכתב `PROJECT-STATUS.md` בתיקיית הפרויקט: המפרט המלא, צ'ק-ליסט שלבים, החלטות (צבע, טבלאות, מי מורשה), כתובת פרויקט Supabase (לא מפתחות!), מזהה אתר Netlify + URL חי, ושורת "הצעד הבא".
- מתעדכן אחרי **כל** תת-שלב.
- בהפעלת הסקיל: קודם מחפשים קובץ מצב קיים. נמצא? — "מצאתי פרויקט קיים — עצרנו בשלב X, ממשיכים?" וקופצים לצעד הרשום.
- כל ה-SQL נשמר ל-`supabase/schema.sql` אידמפוטנטי (בטוח להרצה חוזרת) **לפני** ההרצה; שינויים מאוחרים ל-`supabase/migrations/`. בסיס הנתונים תמיד ניתן לשחזור.

### חוק שתי הנפילות — מסלולי חלופה מוכנים מראש

| נכשל פעמיים                    | עוברים ל־                                                        |
| ------------------------------ | ---------------------------------------------------------------- |
| `npm install`                  | ניקוי מלא → `--legacy-peer-deps`                                 |
| Netlify CLI                    | גרירת תיקיית `dist` לדפדפן (drag & drop)                         |
| הדבקת SQL ב-SQL Editor         | יצירת טבלאות דרך ה-Table Editor הוויזואלי, ואז הרצת ההרשאות בלבד |
| עריכה בלולאת המשוב ששברה build | שחזור git לנקודה האחרונה שעבדה, ניסיון מחדש בצעד קטן יותר        |

המסר למשתמש תמיד: "אני עובר לדרך פשוטה יותר — זה קורה, הכול בסדר." בלי דיבוג חי מולו.

### ניהול שרת הפיתוח (שקוף למשתמש)

הרצה ברקע עם לוג לקובץ; בדיקת פורט תפוס (5173→5174, נרשם בקובץ המצב); המתנה ל-200 לפני "תסתכל"; אחרי כל עריכה — סריקת הלוג לשגיאות חדשות לפני "מה דעתך?"; כיבוי לפני ההעלאה.

### הטיית "קודם באוויר"

אחרי סבב עיצוב אחד לכל היותר לפני העלאה, Claude מציע: "קודם נעלה לאוויר, ואז נשפר עיצוב על האפליקציה החיה." לא חוסם אם המשתמש מתעקש — אבל ברירת המחדל היא לחצות את קו הסיום.

## 5. אבטחה — סגור כברירת מחדל (הנחיית טל)

### התחברות תמיד, בשיטת "רשימת מוזמנים"

- **כל אפליקציה נבנית עם מסך התחברות.** אין מסלול "בלי התחברות" לאפליקציה עם נתוני עסק.
- באפיון נשאל: **"מי צריך גישה לאפליקציה? תן לי את המיילים — שלך ושל הצוות."**
- החשבונות נוצרים יחד עם המשתמש דרך דשבורד Supabase (את הסיסמאות מקליד רק הוא — עצירת חובה).
- אחרי יצירת החשבונות **נועלים את ההרשמה** (כיבוי "Allow new users to sign up"): "נעלתי את הדלת — רק מי שהגדרת נכנס."
- הרשמה פתוחה — רק אם המפרט דורש במפורש שזרים יירשמו (ואז עם אזהרה מפורשת בעברית שנרשמת במפרט).
- מסך ההתחברות מעוצב ומקצועי (חלק מקונבנציות העיצוב), עם הודעות שגיאה בעברית.

### תבניות הרשאות (RLS) מדויקות

- מחיקת שורת ה"allow all for anon" של המקור — החור המרכזי שלו.
- תבניות SQL להעתקה-כלשונה לפי תרחיש:
  - **נתונים אישיים למשתמש** — `user_id` עם ברירת מחדל `auth.uid()`, ארבע policies על `auth.uid() = user_id`
  - **נתונים משותפים לצוות** — גישה ל-authenticated בלבד, לעולם לא ל-anon
  - **תוכן ציבורי לקריאה** — SELECT בלבד ל-anon
  - **טופס פנייה ציבורי** — INSERT בלבד ל-anon, עם בדיקות אורך
- **anon לעולם לא מקבל UPDATE/DELETE.**
- **זיהוי "ריק שקט":** הוספה שהצליחה אבל השורה לא מופיעה בקריאה הבאה = חסימת RLS. Claude אומר "בסיס הנתונים חוסם קריאה — אני מתקן את ההרשאות" ובודק policies — לעולם לא מדווח "הרשימה ריקה".

### היגיינת מפתחות

- בעת הדבקת המפתחות (שלב B3): אימות צורת ה-URL, פענוח ה-JWT ווידוא `role=anon`; אם הודבק מפתח service_role — סירוב: "זה מפתח-על שעוקף את כל ההגנות — נשתמש רק במפתח הציבורי." בדיקת חיבור חיה מיד.
- `.gitignore` שמכסה `.env`/`node_modules`/`dist` נכתב **ראשון**; לפני כל commit — `git check-ignore .env` חובה.
- אף מפתח לא מודפס לצ'אט או לקובץ מצב.

### שער אבטחה E0 (לפני כל העלאה) — בזיקוק Trail of Bits

Claude מריץ אוטומטית ומדווח כצ'ק-ליסט בעברית ("✔ ניסיתי לפרוץ מבחוץ — נחסם"):

1. RLS פעיל על **כל** טבלה, אף policy לא נותנת כתיבה ל-anon.
2. בדיקות "תוקף": קריאה/כתיבה עם מפתח anon חשוף — טבלאות מוגנות חייבות להחזיר ריק/401, כתיבה חייבת להיחסם.
3. אין מפתחות בקוד המקור או ב-dist; `.env` לא ב-git ולא בהיסטוריה.
4. אין ברירות מחדל שנכשלות-פתוח (זיקוק `insecure-defaults`): אין `|| 'ערך קבוע'` על סודות, אין מצב debug, אין CORS פרוץ.
5. אם יש auth — כתובות ה-redirect מוגבלות ל-localhost + דומיין Netlify בלבד, וההרשמה נעולה.

**אסור להעלות עם שער נכשל.**

## 6. עיצוב וּוואו — קונבנציות שclaude מבצע אוטומטית

### עיצוב (בזיקוק impeccable + הנחיות מקוריות)

- **צבע מותג אחד → מערכת שלמה:** באפיון שאלה אחת עם ~6 צבעים בעברית; Claude מייצר בלוק `@theme` של Tailwind v4: סקאלת primary, אפורים מגוונים בגוון המותג, צבעים סמנטיים (הצלחה/אזהרה/סכנה). **אסור** להשתמש בצבעים גולמיים (`bg-blue-500`) — רק בטוקנים.
- **אנטי-דפוסים (מזוקק מ-impeccable):** בלי אפור/שחור טהור (תמיד גוון), בלי טקסט אפור על רקע צבעוני, בלי כרטיס-בתוך-כרטיס, בלי גרדיאנטים סגולים גנריים, בלי אנימציות bounce.
- **טיפוגרפיה עברית:** `@fontsource-variable/heebo` בהתקנה עצמית (לא CDN); סקאלה קבועה של 4 רמות; `tabular-nums` על מספרים; `line-height` 1.6.
- **RTL by construction:** `dir="rtl" lang="he"` על השורש; **אסורות** מחלקות פיזיות (`ml-`/`mr-`/`pl-`/`pr-`/`left-`/`right-`/`text-left`/`text-right`) — רק לוגיות (`ms-`/`me-`/`ps-`/`pe-`/`start-`/`end-`/`text-start`/`text-end`); אייקונים כיווניים מקבלים `rtl:-scale-x-100`; שדות טלפון/מייל/URL מקבלים `dir="ltr"` תחת תווית RTL.
- **מגן Tailwind v4:** רק `@tailwindcss/vite` + `@import "tailwindcss"`; **אסור** `tailwind.config.js`, `@tailwind base/components/utilities`, `npx tailwindcss init` (דפוסי v3 שנכשלים בשקט). השער C→D מוודא שמחלקה אכן מוחלת.
- **פורמט ישראלי:** קובץ `src/lib/format.js` עם Intl he-IL — תאריכים בעברית, ₪ בלי אגורות, זמן יחסי ("לפני 3 ימים"). אף עמוד לא מציג ISO גולמי או מספר גולמי ככסף.

### וואו

- **כפתור וואטסאפ:** כל טבלה עם שדה טלפון מקבלת כפתור וואטסאפ ירוק בכל שורה/כרטיס; `src/lib/whatsapp.js` עם נרמול ישראלי (05→9725) — בלעדיו הלינקים נשברים בשקט; ברכה מוכנה עם שם הלקוח.
- **מסך בית = דשבורד מדדים:** באפיון נשאל "מה שלושת המספרים שאתה רוצה לראות כל בוקר?"; דף הבית נפתח ב-3–4 כרטיסי מדדים חיים + גרף אחד (recharts) איפה שיש תאריכים/סטטוסים; אזור הגרף עטוף `dir="ltr"` (recharts נשבר תחת RTL); כרטיסים לא קורסים על טבלה ריקה — מציגים 0.
- **נתוני דמו בעברית:** אחרי יצירת הטבלאות — 6–10 שורות ריאליסטיות לכל טבלה (שמות ישראליים, טלפוני 05X, סטטוסים מעורבים) עם דגל `is_demo`; נשמר ל-`supabase/seed.sql`; "תמחק את נתוני הדוגמה" מנקה במשפט אחד. בטבלאות עם הרשאות לפי משתמש, שורות הדמו משויכות לחשבון של בעל העסק — אחרת לא ייראו במסך.
- **חוויות קטנות:** skeleton loading במקום ספינר, toast בעברית על כל פעולת CRUD, empty states עם הנחיה, מסך התחברות מלוטש.

### וו רך ל-impeccable

שתי שורות בסקיל: אם סקיל `impeccable` מותקן אצל המשתמש — להריץ את בדיקת העיצוב שלו אחרי שלב הבדיקה המקומית ולאמץ את הממצאים. אם לא — הקונבנציות המובנות מספיקות. אפס תלות.
